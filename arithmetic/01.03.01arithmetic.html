Question: Suppose you have 17 baskets each with 5 apples; or suppose you travel at 17 miles per hour for 5 hours; or suppose that for every egg used in a recipe you include 17 teaspoons of sugar--and you use 5 eggs. Determine the number of apples in all baskets, number of miles traveled, and number of number of teaspoons of sugar.

Answer: Of course each of these involves repeated addition if you want to solve it that way. But repeated addition is common enough that we study it, and this is called multiplication. So the answer to the first question (and each of the others) is 17*5 or 17 added to itself five times. We can make a table algorithm for computing this, although unlike addition, the operation won't go digit-wise like it did for addition. If multiplication went digit-wise we would find 17*5 would equal 1,35 and we could move the 3 over to get 45. But that's wrong, of course. So what's the algorithm?

First we establish the vocabulary that x*y is the product and each component, x and y, is called a factor. When confusion isn't possible, we use concatenation to represent multiplication, so that xy = x*y. Of course we can't do this with numbers since 23 would be ambiguous in an unresolvable way: Is it 2 times 3 or the number 23? But since 2x cannot be mistaken, we will use it to express 2 times x.

As usual, for one-digit operations like 7*5 you just memorize these. Making a multiplication table and memorizing the cells of the table can be (slightly) helpful for this.

For multiple-digit operations, we split things up by digits. Consider 17*5 = (10+7)*5 = 50+35 = 85. If we presented this in a table we would write

Exercise: Given that multiplication is distributive, explain why 2*3*4 is not the same as 2*3 * 2*4.

Answer: Multiplication isn't really "distributive" per se, it's distributive over addition. Since the operation here is not addition, but multiplication throughout, then you only compute this one-product-at-a-time. Of course, associativity tells us that it doesn't matter which product we start with first. We could say that this computes to 6*4 = 24 or we could say it computes to 2*12 = 24. Exercise: Find a number which, when multiplied with 6, produces 192.

Answer: Of course you can find it by cycling through enough numbers until you reach 32. This question is fundamentally equivalent to the equation 6x = 192 and it's also equivalent to 192 divided by 6. Both of these perspectives mirror what we said about subtraction as the inverse of addition.

Again, vocabulary: x/y we call the quotient, where x is the numerator and y is the denominator.

Anyway, as usual, we would like to devise a table algorithm for division. Again we can proceed like with subtraction: Pick the largest digit that doesn't go too far. In this case it's easy to see 6*10 = 60 is too small but 6*100 = 600 is too large, so we start at 6*10. From there we increment the digit until we go too far again. That means checking 6*20 = 120 and 6*30 = 180 and 6*40 = 240 which is too far, so we know the digit 3 is neither too big nor too small, but of course 30 can't be the answer since we're still off by 192-180 = 12. It may be obvious from this stage that the ones-digit should be 2. And here we have a guide for stating the algorithm: Find the largest number starting with 1 and followed only by 0s which you can multiply the denominator by, without getting something larger than the numerator. Then increment the 1 in that digit place until again you get a number too large. Whatever digit that is, record it, then compute the product, subtract from the numerator, and compute the division yet again. Since at every stage we find the infimum digit, the algorithm must be correct. We summarize the process in a table like so.

Let's write a script in which we have a list of numbers, and we want to loop over this list, counting the number of entries that are even. Since even numbers are just those that have 0 remainder when divided by 2, we can use this as a test for whether something is even. Explain what the following code does.

1| exampleList = [2, 3, 4, 5, 6]

2| ag = 0

3| for num in exampleList:

4| if num%2 == 0:

5| ag += 1

Answer:

We start with an example list, and we want to count how many even numbers it has. In this example clearly the answer is 3. We also start an aggregator variable, `ag`, to store the count as we march through the list. Using the for-loop, we call each number in the list `num` and test whether `num%2 == 0`. This effectively tests whether the number is even. For instance when `num` takes the value 2 from the list, the answer is yes, and the code on line 5 will run, therefore incrementing `ag`. If `num` takes the value 3 then the test `num%2 == 0` will test false, and the code on line 5 will not run.

Thus the aggregator variable increases once for each even number in the list, and we have the desired count.

Exercise: We begin with a few definitions.

First, we just have vocabulary over-load. We say that a number z has factor x to mean, as we have said before, that there is a number y such that z = xy. This is nothing new. What's new is that we also commonly describe this by saying that x divides z. We often write this as x|z. Notice that the | symbol is not an operation. The result is not a number. In fact, saying x|z is a statement--it makes a claim. You could regard this as "returning" a truth-value if you like. But it means the same thing as saying "z has x as a factor".

For my own self-amusement I'll define a factor of x to be a stupid factor if that factor is either 1 or x. Thus the stupid factors of 13 are 1 and 13. The stupid factors of 10 are 1 and 10. (Read: "Yeah, they're factors, but they're stupid factors. We want more interesting factors.")

Finally, we define a prime number as any number greater than 1 which has only stupid factors. Any number that's not prime is composite.

(The only reason to exclude 1 is to make certain statements easier later on. Put another way, the number 1 is just too special to group it in with the other primes, so we deliberately exclude it.)

Exercise: The following code will test whether `n` stores a prime number or not. In the given code `n` stores 7 and so will print `True`, but if `n` is changed to 8 for instance, the code will then print `False`. Explain how the code works.

1| n = 7

2| hasSmartFactor = False

3| for num in range(n):

4| if n%(num+1) == 0 and num > 1 and num < n:

5| hasSmartFactor = True

6| print( not hasSmartFactor )

Answer: Line 2 creates a kind of initial hypothesis--since we haven't yet found a smart factor (i.e. a non-stupid factor) then we assume it has none at all. But this is just an initial hypothesis, to be revised if needed when we find the appropriate evidence.

To search for the appropriate evidence, we divide `n` by all numbers 2 through `n`-1. To do that, we make a for-loop ranging through `n` using the variable `num` to store the denominator for each test. Unfortunately in Python this starts at 0 and, if we tried to compute `n%num` we would get a divide-by-zero error. Thus In change this to `n%(num+1)`. But also we're not interested in stupid factors like 1, so I include the caveat `and num > 1` to make sure we are only testing for smart divisors. For the same reason I also include the caveat `num < n`.

If `n%(num+1) and n > 1 and num < n` ever evaluates to `True` then I want to conclude that the number has a smart factor, so in this case I set the variable `hasSmartFactor` to True. At the end of the day, if `hasSmartVariable` stores `True` then the number is not prime and I print `False`; and vice versa. Hence I print the negation of `hasSmartFactor`.

You might find it helpful to insert a print-line within the for-loop to see what's going on as the code runs, rather than only seeing a print-out when the code terminates. If you would like to, you can run the following code and try to trace out why you're getting the print-out that you get.

Exercise: The following code also tests for whether `n` is prime but uses a fairly different method. Explain how this works.

1| n = 7

2| ag = False

3| for num in range(n-2):

4| ag = ag or n%(num+2)

5| print( not ag )

Answer: I create a kind of aggregator variable, but this one aggregates truth-value.

Inside the body of the for-loop I use `num+2` so that I never consider the cases of 0 or 1. Because of this, I can also make the for-loop range from 0 to n-2 and therefore `num` will never take the value `n`. So in total, doing things this way, I will only ever attempt to divide by 2 through `n`-1. If this is confusing you can just run the code

1| n = 7

2| for i in range(n-2):

3| print(i+2)

to see the divisors we will be testing out.

Now, the way that the aggregator variable works is, as I test every possible divisor, if it is not an actual divisor then `ag` will store `ag or False`. Since `ag` starts out as `False` then if it only ever encounters `False` values it will stay `False` through the run of the program. However, if it ever encounters a `True` value then it will store the equivalent of `ag or True` and therefore become true.

You might wonder: But what if it encounters a `False` on the next loop? Will it over-ride the previous `True` value? No! This is the nature of the `or` operator, that if just one value is true the whole thing stays true! So nothing will change its value away from `True` which is just how we want it.

At the end, the `ag` variable indicates whether there was any number from 2 to `n`-1 that divided `n`. If there is such a number we want to print `False` to indicate that `n` is prime; otherwise we want to print `True`.

This shows us what is called the prime factorization of a number. To formally define this we first define what the multiplicity of a prime number is.

If n is a number divisible by prime p, then we say that the multiplicity of p in n is the number of times p divides n. That is to say, if you divide n by p and then divide again, and again, and so on, the multiplicity of p is the number of these iterations you can perform without having a remainder.

Now we can say what the prime factorization of a number is. The prime factorization of n is the list of all the primes that divide n and their multiplicity.

Thus the prime factorization of 1470 can be represented as $$1470 = 2\ast 3\ast 5\ast 7\ast 7$$

This exhibits that 2 is a factor with multiplicity 1, and 3 is a factor with multiplicity 1, and 5 is a factor with multiplicity 1, and 7 is a factor with multiplicity 2.

Exercise: One nice property of a prime number--in spite of the fact that I've called all its factors stupid--is that it doesn't share a (smart) factor with any other number. What makes this so nice is that, if you divide any number by a prime, like for instance 75 divided by 5, then no matter how you split up 75 into factors, 5 will divide one of those factors. So for instance if we split up 75 into 3*25 then 5|25. If we split up 75 into 5*15 then 5|5 and 5|15. In every case, 5 divides one of those numbers. This is unlike 15 which does divide 75, but if we split up 75 into 3*25 then neither 15|3 nor 15|25. In a sense the "cause" of this is the fact that 15 has both a factor of 3 and a factor of 5, and we chose to split of 75 so as to keep the factors of 3 in one place and the factors of 5 in another.

You could get the same result using 6 which divides 36. Since 6 has factors 2 and 3, if we push the factors of 2 to the left and push the factors of 3 to the right we come up with 6|36 and 36 = 4*9 and but neither 6|4 nor 6|9. However, if we had used 3 instead of 6, then no matter how we split 36 into factors, 3 will always divide at least one of those factors. This fact will be useful throughout the rest of Number Theory, especially when we prove the Fundamental Theorem of Arithmetic. Most topics have a "fundamental" theorem. In Arithmetic the fundamental theorem is that every number has a unique prime decomposition, and we'll prove this rigorously when we study Number Theory. Just to give a flavor for what a fundamental theorem is, the Fundamental Theorem of Algebra is (to put it very roughly) an equation with power 5, like $$x^5-3x^4+1=0$$

has 5 solutions. In Calculus the fundamental theorem states a key relationship between the derivative and integral, the two most important objects in Calculus. A fundamental theorem typically tells us some key fact about a key object within the topic.

Anyway, the point is that we will use this nice feature of primes when we prove the Fundamental Theorem of Arithmetic. But although prime numbers have this property, they're no the only ones. There is a notion of numbers being "coprime" where the numbers may not be prime themselves, but in a sense, from the perspective of each other, they behave as if they were prime for each other. Coprime numbers are those which share no common factor. Thus for example 2 and 15 are coprime. So are 36 and 121.

If a number m divides n, and n has factors ab, and m and a are coprime, then m divides b. For example, 2 and 15 are coprime. 2 divides 60. 60 can be split into factors 15 and 4. Therefore 2 must divide 4. In this example I used m=2, n=60, a=15, b=4. The example probably seems kind of uninteresting and contrived, but that's because I'm keeping it simple. It will take some time before the utility of the concept becomes clear.

Exercise: Finally we introduce the concept of the least common multiple, or lcm, for two numbers. Of course we say that m is a multiple of n if there exists a number q such that m = nq. (I.e. m is a multiple of n if you can multiply n by something to get m. E.g. 75 is a multiple of 25 because 75 = 25*3.)

Definition: The least common multiple of m and n is that number a such that there exist numbers b and c such that a=mb and a=nc. For example if m=22 and n=16 then their least common multiple is 176. That is because 176 = 22*8 and also 176 = 16*11. Said another way, the lcm of two number is the least number that is a multiple of both.

Find lcm(5,7) and lcm(15,7) and lcm(35,7). Also explain how to use the prime factorization of two numbers, like 1470 and 55055, to easily obtain their lcm. Also explain the relationship between the lcm and gcd. Also write a script that will find the lcm of two numbers.

Answer: The three lcms are 35, 35, and 35.

To find the lcm of 1470 = 2*3*5*7*7 and 55055 = 5*7*11*11*13 we need a number which is a multiple of 1470 and therefore has all of its factors. So to begin with we write 2*3*5*7*7. This however is not a multiple of 55055 because it's missing some factors, namely two factors of 11 and one factor of 13. Since it already has factors 5 and 7 we need not worry about these. We just add on the factors we need (so that the result is the least value possible) and get 2*3*5*7*7*11*11*13. This computes to 2312310, hence lcm(1470, 55055) = 2312310.

The relationship between the lcm(m,n) = mn/gcd(m,n). If this isn't clear to you, example the example of 1470 and 55055. By multiplying them you would get all of their factors and the factors that they share, the gcd, you would get twice. Therefore if you divide out by the factors that they share, the gcd, they would appear only once just like in the lcm.
